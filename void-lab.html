Void Lab — due politiche di memoria, stesso agente, stesso mondo

Ipotesi: una memoria relazionale A = novelty-gated (scrive solo novità strutturata, cap 200) dà più potere gestionale con meno storage di una B = store-all (scrive ogni percetto). A e B differiscono in una cosa sola: cosa entra in memoria. La sorpresa è calcolata con la stessa identica funzione. Verdetto numerico appaiato, deterministico (mulberry32), falsificabile.

Come si usa
Apri il lab con avvia-lab.bat (doppio click): serve a dare al browser la persistenza vera (OPFS) così gli esperimenti restano salvati per sempre. Se apri il file con un doppio click diretto, il banco funziona lo stesso ma i dati NON sopravvivono alla chiusura (SQLite in-memory).
· Banco: imposti il confronto A/B e premi Run / Sweep — motore deterministico, stessi numeri.
· Archivio: ogni Run/Sweep finisce nel database, con Rifai / Apri / Elimina / Note, filtri, esporta e importa.
· Analisi: query SQL analitiche (DuckDB) su tutti gli esperimenti salvati.

Il motore è DuckDB-WASM (analitico, tipo-ClickHouse). Carica/aggiorna dati copia i run salvati da SQLite dentro DuckDB (runs_olap / metrics_olap): ri-premi dopo nuovi Run/Sweep. Le metriche stanno in metrics_olap come coppie key/value (steps, memSize, reached, finalDist); i parametri (mem A/B, goalPull, noise, seed) si estraggono dai tag del run con regexp_extract. Errori SQL mostrati in chiaro senza rompere la pagina.
