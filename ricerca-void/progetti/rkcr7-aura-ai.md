## Cosa fa
Aura è un assistente AI "invisibile" che vive come overlay sopra lo schermo durante colloqui, esami e meeting: ascolta la conversazione in tempo reale (STT Deepgram), genera risposte-coach con LLM velocissimi (Cerebras, Groq, Gemini) e permette di catturare screenshot di un problema per risolverlo con un modello Vision. Il punto centrale non è la qualità delle risposte ma l'occultamento: la finestra è esclusa dalla cattura schermo (`WDA_EXCLUDEFROMCAPTURE`), non compare nella taskbar, e non guadagna mai focus, così da non far scattare i warning dei software di proctoring. È pensato come strumento a costo zero: porti le tue API key gratuite. Ha un branch Windows maturo e un port macOS in corso.

## Come è fatto
Architettura ibrida: backend Python (FastAPI + WebSocket) che orchestra i servizi (`stt_service`, `vision_service`, `llm_service`, `context_manager`) e un frontend web (HTML/JS in `web/`) renderizzato dentro una finestra nativa gestita da `window_manager.py`. La comunicazione col frontend è via WebSocket con streaming markdown token-per-token (parser custom `streaming-markdown-parser.js`) per far apparire le risposte prima che l'interlocutore finisca di parlare. Lo stealth è la parte più curata: "Ghost Mode" click-through, visibilità senza focus via `SW_SHOWNOACTIVATE` invece di `SW_SHOW`, esclusione dalla cattura schermo, e controllo totale via hotkey globali di sistema (`Alt+...`), così non serve mai cliccare la finestra. Multi-provider con failover e switch a caldo tra modello primario/secondario.

## Cosa possiamo notare di utile per noi
Questo repo è il caso-scuola più completo di **occultamento a livello di sistema operativo** applicato a un agente sempre-presente. La distinzione chiave che loro fanno esplicita — *evitare il focus change* usando `SW_SHOWNOACTIVATE` — è concettualmente identica al problema di un processo/agente che deve osservare e agire su un ambiente senza "disturbarlo" o lasciare tracce osservabili: la stealth qui è definita come *non alterare le variabili che l'osservatore monitora* (focus, taskbar, framebuffer catturato). È una definizione operativa di occultamento molto pulita, trasferibile a chi ragiona su steganografia di stato. Il `PROCTORING_STEALTH_GUIDE.md` è di fatto una tassonomia dei canali di rilevamento (focus, tab, recording, window detection, mouse) con la contromisura per canale — utile come checklist mentale per pensare "quali canali laterali rivelano lo stato interno di un sistema che vuole restare nascosto". Dove diverge dal lavoro sul vuoto/particelle: qui non c'è world-model né memoria novelty-gated, è puro I/O reattivo stateless; il `context_manager` è una memoria di sessione lineare, non gated. Ma lo streaming markdown token-per-token come "oscilloscopio" sullo stato di generazione dell'LLM è un pattern riusabile per visualizzare in tempo reale uno stato interno che evolve.

## Da rubare
1. **Tassonomia dei canali di rivelazione + contromisura-per-canale** (dal guide di proctoring): riusarla come framework per ragionare su cosa rende osservabile lo stato interno di un sistema che deve restare occulto.
2. **Pattern "visibilità senza focus"** (`SW_SHOWNOACTIVATE`): mostrare/aggiornare un pannello di monitoring — es. un oscilloscopio su stati interni — senza mai rubare il focus né alterare l'ambiente osservato.
3. **Streaming markdown parser token-per-token via WebSocket**: componente pronto per rendere in tempo reale l'evoluzione di uno stato che si "srotola", utile per gli oscilloscopi sugli stati interni.
